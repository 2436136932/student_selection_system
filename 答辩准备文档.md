## 一、系统概述

### 1.1 项目背景与目标

**项目名称：** 学生评奖评优资格筛选系统  
**项目定位：** 面向高校的多角色（学生/教师/管理员）奖项申请与审批一体化平台。

**核心功能矩阵：**

| 模块 | 功能 | 涉及角色 |
|------|------|----------|
| 认证授权 | 登录/注册/密码重置/角色权限 | 全部 |
| 课程管理 | 课程CRUD/选课/退课 | 管理员/学生 |
| 成绩管理 | 成绩录入/查询 | 教师/管理员/学生 |
| 奖项管理 | 奖项发布/状态流转 | 管理员 |
| 奖项申请 | 学生申请/材料上传/二级审批 | 学生/教师/管理员 |
| 获奖记录 | 历史存档/获奖公示 | 全部 |
| AI智能推荐 | 五维度打分/TOP10推荐 | 学生/管理员 |
| 实时聊天 | WebSocket点对点通信 | 全部 |
| 数据统计 | ECharts可视化驾驶舱 | 全部 |
| 通知公告 | 通知发布/轮播图管理 | 管理员/全部 |

### 1.2 技术栈总览

**前端层：** Vue 3 (Composition API + `<script setup>`) + Vite 5 + Element Plus 2.4 + ECharts 6 + Vue Router 4 + Axios + SortableJS + WebSocket (自动重连)

**后端层：** Java 21 + Spring Boot 3.1.5 + MyBatis-Plus 3.5.3 (LambdaQueryWrapper) + Spring Security + JWT (jjwt 0.12.3, HMAC-SHA256)

**数据层：** MySQL 8.0 (14 张表: users/students/teachers/courses/scores/selections/awards/student_award_applications/student_award_records/notices/carousels/majors/chat_sessions/chat_messages/recommendation_weights/verification_codes/competition/roles/permissions)

**工具：** Apache POI 5.2.3 (获奖记录 Excel 导出), QQ Mail SMTP (邮件通知), Spring WebSocket (实时聊天), Lombok 1.18.34

### 1.3 核心实体关系

- `User` (1) ↔ (0..1) `Student` / `Teacher`（用户与师生资料一对一）
- `Student` (1) ↔ (*) `Selection` ↔ (1) `Course`（选课，多对多关联）
- `Student` (1) ↔ (*) `Score` ↔ (1) `Course`（成绩）
- `Student` (1) ↔ (*) `StudentAwardApplication` ↔ (1) `Award`（奖项申请）
- `Student` (1) ↔ (*) `StudentAwardRecord` ↔ (1) `Award`（获奖记录，反范式快照）
- `User` (1) ↔ (*) `ChatSession` ↔ (*) `ChatMessage`（聊天）

### 1.4 核心业务流

管理员发布奖项 → 学生申请并上传材料 → 教师审批（通过/拒绝） → 管理员最终审批（通过则生成获奖记录） → 邮件通知获奖学生

### 2.2 菜单与权限体系（`App.vue`）

`App.vue` 是系统的外壳组件，管理整个布局。菜单权限通过 `getVisibleMenus` 计算属性实现：

```js
// 菜单定义（menuItems 是普通数组，每个菜单项带 roles 权限标记）
const menuItems = [
  { index: '/students', icon: User, text: '学生管理', roles: ['admin', 'teacher'] },
  { index: '/teachers', icon: Briefcase, text: '教师管理', roles: ['admin'] },
  // ...其他菜单
]

// 根据用户角色动态过滤可见菜单 + 支持自定义排序
const getVisibleMenus = computed(() => {
  const currentRole = userInfo.value.role
  const filteredMenus = menuItems.filter(menu => menu.roles.includes(currentRole))

  // 从 localStorage 读取用户自定义的菜单顺序
  const savedMenuOrder = localStorage.getItem('menuOrder')
  if (savedMenuOrder) {
    // 按保存的顺序重新排列
    const menuMap = new Map(filteredMenus.map(m => [m.index, m]))
    const ordered = []
    JSON.parse(savedMenuOrder).forEach(index => {
      if (menuMap.has(index)) ordered.push(menuMap.get(index))
      menuMap.delete(index)
    })
    menuMap.forEach(m => ordered.push(m))
    return ordered
  }
  return filteredMenus
})
```

### 3.4 AI 智能推荐算法（`AwardRecommendationService.java`）

#### 算法流程

输入 `studentId` → 查询学生信息（专业/成绩/历史获奖）→ 获取所有"已发布且进行中"的奖项 → 排除已申请的 → 对每个可用奖项计算五维度加权得分 → 按总分降序排列 → 返回 Top 10 推荐列表

#### 五维度得分计算

**① 成绩匹配度（默认权重 40%）**

```java
// AwardRecommendationService.java:259-286 — 成绩匹配度计算
private double calculateGradeScore(List<Score> studentScores) {
    if (studentScores == null || studentScores.isEmpty()) {
        return 0.0;
    }
    double totalScore = 0.0;
    double totalCredits = 0.0;
    for (Score score : studentScores) {
        if (score.getTotalScore() != null) {
            totalScore += score.getTotalScore().doubleValue();
            totalCredits += 1.0;
        }
    }
    if (totalCredits == 0) return 0.0;
    double avgScore = totalScore / totalCredits;
    if (avgScore >= 90) return 100.0;
    if (avgScore >= 85) return 90.0;
    if (avgScore >= 80) return 80.0;
    if (avgScore >= 75) return 70.0;
    if (avgScore >= 70) return 60.0;
    return 50.0;
}
```

**② 已有奖项匹配度（默认权重 30%）**

根据目标奖项的级别（national/provincial/school/department），结合学生已有最高级别奖项打分：
- 目标是国家奖 → 学生有国家奖=100分，有省级奖=80分，有校级奖=60分……
- 目标是省级奖 → 学生有省级或国家奖=100分……

**③ 专业匹配度（默认权重 15%）**

```java
// AwardRecommendationService.java:341-352 — 专业匹配度计算
private double calculateMajorScore(Student student, Award award) {
    String requirement = award.getRequirement();
    if (requirement == null) return 100.0;
    if (requirement.contains(student.getMajor())) return 100.0;
    return 70.0;
}
```

**④ 历史数据（默认权重 10%）**

该奖项的历史申请人数越多，得分越高（说明该奖项越受关注，数据越可靠）：
- 申请数=0 → 50分
- 申请数<10 → 70分
- 申请数<30 → 85分
- 否则 → 100分

**⑤ 竞争程度（默认权重 5%）**

申请人数与名额的比例越低，竞争越小，得分越高：
- ratio < 1.0 → 100分（名额充足）
- ratio < 2.0 → 80分
- ratio < 3.0 → 60分
- ratio < 5.0 → 40分
- 否则 → 20分

**最终得分 = Σ(维度原始分 × 维度权重)**

**获奖概率 = (总匹配度/100) × 竞争因子 × 100**

其中竞争因子在申请人数超过名额时按比例衰减（最低 0.3）。

**推荐语生成规则：**

| 匹配度 | 获奖概率 | 推荐语 |
|--------|---------|--------|
| ≥85 | ≥70 | 强烈推荐！条件非常符合，获奖概率很高 |
| ≥70 | ≥50 | 推荐申请！具备较好竞争力 |
| ≥60 | - | 可以尝试！竞争可能较激烈 |
| <60 | - | 谨慎考虑！建议先提升自身条件 |

### 3.5 奖项生命周期自动管理（`AwardStatusScheduler.java`）

**定时策略：** `@Scheduled(cron = "*/10 * * * * ?")` 每 10 秒扫描一次（开发/演示环境，生产可调低频率）

**状态流转规则：** `status`（手动设置）: 未发布 → 已发布 → 已结束。`current_status`（自动计算）: 未发布→待开始, 已发布+未到截止→进行中, 已发布+已到截止→已关闭(同时status改为已结束), 已结束+已到截止→已关闭

**邮件通知：** 奖项结束时扫描 `student_award_records` 表，给每个获奖学生发送 QQ 邮件，用 `email_sent` 字段防止重复发送。

### 3.6 WebSocket 实时聊天

**服务端（`ChatWebSocketHandler.java`）：**
- `ConcurrentHashMap<String, WebSocketSession>` 维护在线用户会话
- 连接时解析 URL 参数 `?userId=&token=`，验证 JWT 后才建立连接
- 收到消息 → 保存到数据库 → 转发给接收方
- 提供 `getOnlineUserCount()` 和 `getOnlineUserIds()` API

**客户端（`websocketService.js`）：**
- 自动连接 + 5 秒间隔重连
- 消息序列化为 JSON
- 断线检测（`onclose` 事件触发重连定时器）

### 3.7 文件上传与下载

**上传流程（`StudentAwardApplicationController.uploadMaterial()`）：** 客户端上传文件 → 校验扩展名白名单 → 校验 ≤10MB → 验证用户身份（必须是学生）→ UUID 重命名 → 保存到 `uploads/` 目录 → 返回 `{materialPath, materialName, materialSize, materialType}`

**下载/预览：**
- 下载：`Content-Disposition: attachment`，触发浏览器下载
- 预览：`Content-Disposition: inline`，支持浏览器内预览（图片/PDF）
- MIME 类型自动推断（通过文件扩展名映射）

## 四、答辩问题与答案

### 4.1 系统设计类

---

**Q1: 为什么选择 MyBatis-Plus 而不是 Spring Data JPA？**

> **核心原因有两个：**
>
> 第一，**复杂查询需求**。本系统涉及大量统计查询（按状态分组统计、按专业聚合、时间趋势分析），这些查询需要写原生 SQL 才能获得最佳性能。MyBatis-Plus 允许我们在 XML 或注解中手写 SQL，精细控制执行计划。JPA 的 JPQL / Criteria API 在这种场景下要么生成低效 SQL，要么写法比原生 SQL 还复杂。
>
> 第二，**学习成本和团队偏好**。MyBatis-Plus 的 `LambdaQueryWrapper` 提供了链式查询的便利性，比如：
> ```java
> // AwardRecommendationService.java:58-60 — 链式条件查询示例
> new LambdaQueryWrapper<Award>()
>     .eq(Award::getStatus, "已发布")
>     .eq(Award::getCurrentStatus, "进行中")
> ```
> 既直观又类型安全。同时保留了 XML 映射文件对复杂 SQL 的支持。
>
> 从代码中可以看到，我们确实注释掉了 JPA 的依赖（pom.xml 第 38-42 行），明确选择了 MyBatis-Plus。

---

**Q2: JWT 方案为什么比传统 Session 更适合本项目？**

> **三个关键原因：**
>
> 第一，**前后端分离架构**。前端是独立部署的 Vite 应用（端口 5173），后端是 Spring Boot（端口 8080），存在跨域。JWT 通过 `Authorization` 请求头发送，天然支持跨域。而 Session 依赖 Cookie，跨域场景下需要复杂的 CORS + Cookie 配置（`withCredentials: true`），且容易受到 CSRF 攻击。
>
> 第二，**无状态扩展性**。JWT 不需要服务端存储会话信息，部署多实例时无需共享 Session（不需要 Redis/Session 集群），简化了架构。
>
> 第三，**角色信息内嵌**。JWT 的 claims 中直接携带角色信息，后端 JWT 过滤器解析后即可设置权限上下文，省去了每次请求都查库获取角色信息的开销。
>
> 安全性方面：JWT 使用 HMAC-SHA256 签名防篡改，过期时间 2 小时，密码使用 BCrypt 加密。对于校内系统，这个安全级别已经足够。

---

**Q3: 为什么 `student_award_records` 表要冗余存储学生姓名、学号、专业等信息？这不是违反数据库范式吗？**

> 这是典型的"**空间换时间**"的反范式设计，目的有三：
>
> 第一，**历史快照**。获奖记录一旦生成就是历史档案。如果学生后来改了名字、转了专业，已生成的获奖记录不应该跟着变。冗余存储保证了数据的"快照一致性"。
>
> 第二，**查询性能**。获奖公示页面需要展示获奖学生姓名、学号、专业、奖项名称等信息。如果完全依赖 JOIN 查询，每次都要关联 students 和 awards 表，数据量大时性能会下降。冗余存储后一次单表查询即可。
>
> 第三，**数据隔离**。即使 students 表或 awards 表的数据被误删或修改，获奖记录仍然完整可查。
>
> 这是业界在报表系统和归档系统设计中的常见模式。

---

**Q4: 系统的角色权限模型是怎么设计的？如何防止学生越权操作？**

> 系统采用三层权限控制：
>
> **第一层：URL 白名单（SecurityConfig.java）。** 公开接口在 `authorizeHttpRequests` 中配置 `.permitAll()`，其余接口默认需要认证。
>
> **第二层：方法级注解（@PreAuthorize）。** 在 Controller 方法上标注，支持多角色组合：
> ```java
> // 仅管理员 — 如 UserController.java:34, TeacherController.java:37
> @PreAuthorize("hasRole('ADMIN')")
> // 学生 + 教师 + 管理员 — 如大部分查询接口
> @PreAuthorize("hasRole('ADMIN') or hasRole('TEACHER') or hasRole('STUDENT')")
> // 仅学生 — 如 StudentAwardApplicationController.java:79
> @PreAuthorize("hasRole('STUDENT')")
> ```
> 比如学生创建申请的接口是 `@PreAuthorize("hasRole('STUDENT')")`，教师无法调用。
>
> **第三层：业务级权限检查（Controller 方法内部）。** 学生在查询申请时，代码会从 `Principal` 获取当前用户名 → 查出 studentId → 与请求查询的 studentId 比对，不一致则返回 403 Forbidden。教师也只能查看自己负责审批的申请。
>
> 通过这三层防护，可以有效防止越权操作。

---

**Q5: 数据库表之间的关联关系是外键还是逻辑关联？为什么？**

> 项目中既有**外键约束**（如 `selections` 表的 `student_id` 和 `course_id` 使用 `ON DELETE CASCADE`），也有**纯逻辑关联**（如 `StudentAwardRecord` 中的 `student_id` 和 `award_id`）。
>
> 设计原则是：**核心业务数据用外键保证完整性，历史/归档数据用逻辑关联追求灵活性。**
>
> - `selections` 表：学生选课记录，级联删除可以保证退课时的数据一致性。
> - `scores` 表：成绩数据，用外键 + 唯一约束确保一个学生一门课只有一条成绩。
> - `student_award_records` 表：历史记录，逻辑关联避免外键对归档操作的干扰。
>
> MyBatis-Plus 不强制使用外键，我们在业务 Service 层通过条件构造器保证引用完整性。

### 4.2 技术实现类

**Q6: AI 推荐算法是怎么实现的？权重依据是什么？**

> 这是一个**五维加权打分模型**。
>
> 对于每个"已发布且进行中"的奖项，算法计算以下五个维度的得分：
>
> | 维度 | 权重 | 计算逻辑 |
> |------|------|---------|
> | 成绩匹配度 | 40% | 平均分分档：≥90→100, ≥85→90, ≥80→80, ≥75→70, ≥70→60, <70→50 |
> | 已有奖项 | 30% | 根据奖项级别与学生已有最高奖项级别的匹配程度打分 |
> | 专业匹配 | 15% | 奖项要求包含学生专业→100分，否则70分 |
> | 历史热度 | 10% | 奖项历史申请人数越多分越高（数据可靠性越高） |
> | 竞争程度 | 5% | 申请人数/名额比例越低分越高 |
>
> **最终得分 = 40%×成绩分 + 30%×奖项分 + 15%×专业分 + 10%×历史分 + 5%×竞争分**
>
> 权重的设定依据是**领域专家经验**：成绩是评奖的核心指标，所以权重最高（40%）；已有奖项反映学生的综合竞争力（30%）；专业匹配确保奖项与学生相关（15%）；其余作为微调因子。总权重和为 100%。
>
> **权重可调节：** 管理员可以在前端的"权重配置"对话框中通过滑块实时调整，数据持久化在 `recommendation_weights` 表中。调整后立即生效，无需重启服务。
>
> 此外，算法还输出**获奖概率**（匹配度 × 竞争因子）、**竞争程度标签**（竞争较小/适中/激烈/非常激烈）、**推荐语**（根据匹配度和概率的四档提示），以及**诊断信息**（当推荐为空时帮助排查原因）。

**Q7: 如果学生没有成绩，推荐算法会怎么处理？**

> 代码中有明确的防御性处理：
> ```java
> private double calculateGradeScore(List<Score> studentScores) {
>     if (studentScores == null || studentScores.isEmpty()) {
>         return 0.0;  // 无成绩时返回 0 分
>     }
> }
> ```
> 当 `gradeScore = 0`，乘以权重 40% 后，成绩这一维贡献 0 分。虽然其他维度仍可得分，但总分会显著降低。同时，如果该维度的原始分是 0，说明学生暂无成绩，推荐系统也会给出较低的排名。
>
> 在诊断信息中，前端会显示"您暂未录入成绩，请先联系教师录入"，提示学生先完成成绩录入。

**Q8: 奖项状态自动流转的定时任务为什么设置每 10 秒执行一次？生产环境也是这样吗？**

> 这是一个**开发测试用的频率**。`@Scheduled(cron = "*/10 * * * * ?")` 表示每 10 秒执行一次，这样在演示时可以快速看到奖项状态从"进行中"变为"已关闭"。
>
> **生产环境**应该调整频率，比如每小时执行一次：
> ```java
> @Scheduled(cron = "0 0 * * * ?") // 每小时整点执行
> ```
> 或者每天凌晨执行一次：
> ```java
> @Scheduled(cron = "0 0 2 * * ?") // 每天凌晨2点执行
> ```
>
> 定时任务的逻辑本身是**幂等的**——每次执行都会重新计算所有奖项的状态，重复执行不会产生副作用。这也是为什么频率可以设得很高。

**Q9: WebSocket 连接如何做安全认证？**

> 连接时前端在 URL 中附带 token：
> ```
> ws://localhost:8080/ws/chat?userId=123&token=eyJhbGciOiJIUzI1NiJ9...
> ```
> 后端 `ChatWebSocketHandler.afterConnectionEstablished()` 中：
> 1. 从 URL 参数提取 token
> 2. 调用 `jwtUtil.extractUsername(token)` 解析用户名
> 3. 调用 `userDetailsService.loadUserByUsername(username)` 加载用户
> 4. 调用 `jwtUtil.validateToken(token, userDetails)` 验证签名和有效期
> 5. 验证通过后才加入 `ConcurrentHashMap sessions`
> 6. 验证失败则 `session.close()` 拒绝连接
>
> 同时，SecurityConfig 中将 `/ws/chat` 配置为 `permitAll()`，是因为 WebSocket 握手阶段的认证逻辑在 Handler 中自定义实现，不需要 Spring Security 介入。

**Q10: 文件上传的安全性如何保证？**

> 采取了以下措施：
>
> 1. **文件类型白名单：** 仅允许 `docx,doc,pdf,pptx,ppt,png,jpg,jpeg,xls,xlsx`，在配置文件中统一管理（`application.yml` 第 69 行）。后端校验文件扩展名，不匹配直接拒绝。

> 2. **文件大小限制：** 单文件最大 10MB（`spring.servlet.multipart.max-file-size: 10MB`），后端也做二次校验。
>
> 3. **UUID 重命名：** 上传后文件名为 `UUID.扩展名`，防止文件名冲突和路径遍历攻击。
>
> 4. **存储路径隔离：** 文件存储在专用的 `uploads/` 目录，不混入代码目录。
>
> 5. **权限控制：** 只有通过认证的用户才能上传文件；下载和预览也需要认证，且用户只能访问自己有权限的申请材料。

### 4.3 业务场景类

**Q13: 如果教师审批通过后，管理员审批时发现材料不全怎么办？**

> 系统设计上，管理员的审批结果独立于教师的审批结果。管理员可以选择"拒绝"并填写审批意见（如"材料不全，请补充"）。此时申请的 `status` 变为 4（管理员已拒绝），但 `teacher_approval_status` 仍保持 1（通过）。
>
> 学生可以查看拒绝原因，重新上传材料后再次提交申请，此时新的申请会重新进入审批流程。
>
> 需要注意的是，目前系统没有"教师审批后学生不能修改申请"的限制——这是一个可改进点：可以在教师审批通过后锁定申请，不允许修改。

**Q14: 一个奖项名额只有 5 个，但 10 个学生都通过了审批怎么办？**

> 审批通过只代表"资格符合"，不代表一定获奖。最终名额分配在**结果公示阶段**由管理员决定。管理员可以根据：
> - 综合匹配度评分（参考 AI 推荐的分数）
> - 申请材料质量
> - 其他评审标准
> 从通过的学生中选择最终获奖者，生成 `student_award_records` 获奖记录。
>
> 名额字段 `award.quota` 目前作为参考提示，在 AI 推荐的竞争程度计算中会用到，但在审批环节没有硬性限制。这是一个可以优化的方向：在管理员审批时，如果通过人数达到名额上限，自动关闭申请通道。

**Q15: 学生的申请长期处于"待教师审批"状态，系统有催办机制吗？**

> 目前系统**没有自动催办机制**，但提供以下辅助手段：
>
> 1. **实时聊天：** 学生可以通过 WebSocket 聊天功能联系教师，催促审批。
> 2. **状态透明：** 学生可以随时查看申请状态，知道申请卡在哪一步。
> 3. **管理员介入：** 管理员可以在后台查看所有待审批申请，联系教师处理。
>
> 这是一个潜在的改进点：可以增加**超时自动升级**机制——如果教师在 3 天内未审批，系统自动发送邮件提醒；如果 5 天仍未审批，自动将审批权限升级到管理员。

**Q16: 邮件发送失败怎么办？会影响核心流程吗？**

> 邮件发送失败**不影响核心业务流**。`EmailServiceImpl.java:24-35` 内部捕获了异常，不会向上抛出：
> ```java
> // EmailServiceImpl.java:24-35 — 邮件发送内部异常捕获
> public void sendAwardNotificationEmail(...) {
>     try {
>         MimeMessage message = mailSender.createMimeMessage();
>         // ... 构建邮件内容
>         mailSender.send(message);
>     } catch (Exception e) {
>         logger.error("发送邮件失败: {}", e.getMessage());
>         // 不抛出异常，不阻断主流程
>     }
> }
> ```
> `AwardStatusScheduler.java` 中调用 `sendAwardNotificationToStudents()`，该方法内部循环调用 `emailService.sendAwardNotificationEmail()`，即使个别学生邮件发送失败，也不会影响奖项状态流转。
> 即使 QQ 邮箱 SMTP 服务临时不可用，奖项的状态流转和记录生成都不会受影响。系统日志中会记录失败原因，管理员可以检查日志后手动重发。

**Q17: 如果一个学生同时申请多个奖项，系统如何避免重复申请？**

> 双重防护：
>
> 1. **数据库层：** `student_award_applications` 表中没有显式的唯一约束，但 unique 逻辑在业务层实现。
>
> 2. **业务层：** `AwardRecommendationService.recommendAwards()` 中明确过滤：
> ```java
> // AwardRecommendationService.java:135-143 — 跳过已申请的奖项
> boolean alreadyApplied = studentAwardApplicationMapper.exists(
>     new LambdaQueryWrapper<StudentAwardApplication>()
>         .eq(StudentAwardApplication::getStudentId, studentId)
>         .eq(StudentAwardApplication::getAwardId, award.getId())
> );
> if (alreadyApplied) {
>     continue;
> }
> ```
> 创建申请时也做了相同检查。前端在"立即申请"按钮上也做了禁用处理（匹配度 < 50 时不可点击）。

**Q19: 开发过程中遇到的最大技术困难是什么？怎么解决的？**

> **最大困难：AI 推荐算法的权重调优与空推荐问题排查。**
>
> **问题表现：** 初期推荐结果经常为空，但不知道原因——是数据库没有奖项？奖项状态不对？还是学生已申请了所有奖项？
>
> **解决方案：** 我在推荐服务中添加了 `getDiagnostic(Long studentId)` 诊断方法，返回详细的排查信息：
> ```json
> {
>   "totalAwardsInDb": 5,
>   "availableAwards": 2,
>   "alreadyAppliedCount": 2,
>   "scoreCount": 0,
>   "awardDetails": [
>     {"name": "国家奖学金", "status": "已发布", "currentStatus": "进行中", "alreadyApplied": true}
>   ]
> }
> ```
> 前端在推荐为空时展示诊断面板，一眼就能看出是"成绩未录入"还是"已申请所有奖项"。这个诊断工具在测试和演示时非常实用。

**Q21: 项目中有没有使用设计模式？**

> 有，主要体现在以下方面：
>
> 1. **模板方法模式：** `JwtAuthenticationFilter` 继承 `OncePerRequestFilter`，框架定义请求过滤的模板，子类实现具体的过滤逻辑。
>
> 2. **策略模式：** Spring Security 的 `UserDetailsService` 接口，不同的认证策略可以实现不同的 `loadUserByUsername()` 方法。
>
> 3. **代理模式：** MyBatis-Plus 的 Mapper 接口，框架在运行时生成代理类执行 SQL。
>
> 4. **单例模式：** Spring 管理的 Bean（Controller、Service、Mapper）默认都是单例的。
>
> 5. **观察者模式：** WebSocket 的 `onopen`、`onmessage`、`onclose` 回调就是观察者模式的体现。
>
> 6. **工厂模式：** `Jwts.builder()` 创建 JWT token 实例。

**Q22: 项目中的代码有哪些可以优化的地方？**

> 客观来说，存在以下可优化点：
>
> 1. **响应格式统一**：目前有的接口返回 `ResponseEntity<Map>`（手动组装 success/data/message），有的直接返回实体对象。应该统一为全局响应体封装。
>
> 2. **异常处理粒度**：GlobalExceptionHandler 可以更细化，针对不同异常类型（参数校验异常、业务异常、系统异常）返回不同的状态码和消息。
>
> 3. **DTO 与 Entity 分离**：目前 Controller 层直接使用 Entity 接收请求参数，应该引入 DTO（数据传输对象）层，避免敏感字段泄露和参数绑定漏洞。
>
> 4. **SQL 性能优化**：统计接口目前是逐个计数（count 多次调用），可以用一个联合查询优化。
>
> 5. **事务管理**：部分 Service 方法需要添加 `@Transactional` 保证原子性，如创建申请 + 上传材料的组合操作。
>
> 6. **前端代码体积**：`StudentAwardApplicationView.vue` 接近 1500 行，应该拆分为多个子组件。

**Q23: 谈谈对 Vue 3  Composition API 和 Options API 的理解，为什么选 Composition API？**

> Composition API 相比 Options API 的优势：
>
> 1. **更好的逻辑复用**：Options API 通过 mixins 复用逻辑，容易产生命名冲突和来源不明的问题。Composition API 通过 `useXxx()` 函数封装，每个函数职责清晰。
>
> 2. **更好的类型推导**：`<script setup>` + TypeScript 的组合可以提供完整的类型推断。
>
> 3. **更灵活的组织**：Options API 强制按 data/methods/computed 组织代码，相关的逻辑被分散在不同区域。Composition API 允许按功能组织代码，比如与 AI 推荐相关的所有状态和函数放在一起。
>
> 在本项目中，`AwardRecommendationView.vue` 的 `<script setup>` 就是很好的例子——权重相关的变量和函数集中在一起，推荐相关的变量和函数集中在一起，代码逻辑清晰。

**Q24: Spring Boot 3.1.5 相比旧版本有什么新特性用到了吗？**

> Spring Boot 3.x 基于 Spring Framework 6.x，核心变化包括：
>
> 1. **Jakarta EE 命名空间**：从 `javax.*` 迁移到 `jakarta.*`（如 `jakarta.servlet.Filter`、`jakarta.persistence.*`）。这在 pom.xml 中可以看到——我们使用的 `jjwt` 版本 0.12.3 已经支持 Jakarta。
>
> 2. **Java 17+ 基线**：Spring Boot 3 要求 Java 17 以上，本项目使用 Java 21，用到了 Java 21 的 Record 类、Switch 模式匹配等新特性（pom.xml 中开启了 `--enable-preview`）。
>
> 3. **GraalVM 原生镜像支持**：Spring Boot 3 支持编译为 GraalVM 原生镜像，启动时间可降至毫秒级（本项目未使用，但具备扩展能力）。
>
> 4. **Security 配置方式变化**：使用 Lambda DSL 配置安全过滤链（`SecurityConfig.java` 第 82-133 行），比旧版的继承 `WebSecurityConfigurerAdapter` 更现代化。
